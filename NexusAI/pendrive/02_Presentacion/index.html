01 · Punto de partida

Moodle hoy es un
repositorio de archivos,
no un asistente.

Las plataformas LMS son fundamentales en la universidad, pero en la práctica se usan para descargar PDFs. Las consultas reales pasan por otro lado: ChatGPT, WhatsApp, Drive.

Para el alumno
Las dudas van directo al docente o a ChatGPT genérico, sin contexto del material real del curso.
Para el docente
No tiene visibilidad de qué le preguntan a la IA sus alumnos, ni qué temas le faltan al material que subió.
Para la institución
Datos académicos en servicios SaaS externos. Cero soberanía sobre cómo se procesa el contenido.

03 · La solución

NexusAI es un plugin Moodle
con un asistente IA que responde
sobre tu material.

RAG auténtico con citas trazables
Cada respuesta enlaza al fragmento exacto del material del curso. Si la pregunta no se puede responder con el material disponible, el sistema lo admite, no inventa.
Self-hosted, datos en la institución
El patrón Hybrid PHP Proxy garantiza que la API key del LLM nunca llega al navegador. Datos académicos en infraestructura controlada por la institución.
Multi-curso opcional
El alumno puede consultar el material de todos sus cursos a la vez. Las respuestas citan la materia de cada fragmento.
Feedback loop al docente
El sistema registra automáticamente las preguntas que no pudo responder y se las muestra al docente para mejorar el material.

10 · Gestión de riesgos

Identificamos, monitoreamos
y mitigamos.

Trabajamos con una matriz de riesgos viva durante todo el proyecto. Estos son los cuatro que más impactaron — y cómo los neutralizamos.

Alto · Técnico
R-01
LLM inventa respuestas (alucinaciones)
Mitigación: prompts estrictos + threshold de similitud + regex post-respuesta que detecta "no encontré" y dispara feedback al docente.
Alto · Seguridad
R-02
Filtración de API key del LLM
Mitigación: patrón Hybrid PHP Proxy — la clave nunca llega al navegador. Triple defensa: Bearer + HMAC + nonce Redis con TTL.
Medio · Cronograma
R-03
Scope creep entre sprints
Mitigación: backlog congelado al inicio de cada sprint y política explícita de "esto va al post-MVP" sin culpa.
Medio · Operativo
R-04
Caída del backend durante la defensa
Mitigación: backend en Railway con autodeploy + health checks + fallback con capturas pregrabadas de cada paso del demo.

14 · Earned Value Management

Valor ganado del MVP.

Aplicamos EVM al cierre del proyecto para medir cumplimiento real contra lo planificado. Las métricas confirman un MVP entregado en plazo con sobrecosto controlado, sano para un proyecto con scope creep.

Sobrecosto leve (~8%)

SV — Schedule Variance (SP)
10 SP de atraso compensados en S4

CV — Cost Variance

Premisas del cálculo
  • BAC · 265 SP × 1.5 h/SP = 400 h ≈ $10.000 notional
  • Tarifa · $25/h (junior dev de mercado)
  • EV · 255 SP completados = 385 h equivalentes
  • AC · ~420 h reales (incluye retrabajos)
Proyección al cierre
  • EAC · BAC / CPI = ~$10.875
  • VAC · BAC − EAC = −$875
  • ETC · ~$0 (MVP cerrado, sin trabajo pendiente)
  • % Completado · EV/BAC = 96.2%

Lectura: 96% de cumplimiento de cronograma y 92% de eficiencia de costos. Ambos ratios están dentro del rango "verde" del PMI para proyectos ágiles (0.90 - 1.10).
